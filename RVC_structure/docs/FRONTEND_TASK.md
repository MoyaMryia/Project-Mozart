# 前端（预处理端）工作清单

> 本文档面向 **Project Mozart 预处理端开发团队**，说明需要实现的功能、接口调用方式与对接步骤。  
> 后端（RVC 变声）只接收预处理后的契约流，不做任何原始音频处理。

---

## 1. 前端是谁？职责是什么？

在这个架构中，**前端 = 预处理端**，它是用户设备上的一个音频服务，负责：

```
┌────────────────────────────────────────────────────────────┐
│  前端（用户设备）                                           │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │ 麦克风  │─►│ 预处理  │─►│ 契约封装 │─►│ UDP发送 │─────┼──► 后端
│  │ 采集    │  │ 流水线  │  │ 打包    │  │ 18000   │     │   Jetson
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘     │
│                                              ▲            │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │            │
│  │ 扬声器  │◄─│ 播放器  │◄─│ 契约解析 │◄────┘            │
│  │ 输出    │  │ 声卡    │  │ 18000   │                    │
│  └─────────┘  └─────────┘  └─────────┘                    │
└────────────────────────────────────────────────────────────┘
```

你的核心职责：**把麦克风采集的"脏音频"变成"干净的契约流帧"发给后端，再把后端变声后的音频播出来。**

---

## 2. 前端必须实现的 5 大模块

### 2.1 麦克风采集（I/O 层）

**做什么**：从系统声卡抓取原始音频，交给预处理流水线。

**技术要求**：
- 目标格式：48kHz / mono / float32（预处理内部统一用这个）
- 采集方式：
  - **Linux**：PipeWire 或 PulseAudio 的 capture 流
  - **Windows**：WASAPI capture loopback
  - **macOS**：CoreAudio
- 帧大小：每次回调提供 20ms 的样本（48kHz 下 = 960 个 float32）
- 不可阻塞：音频采集回调必须在 <5ms 内返回，不能做耗时操作

**注意事项**：
- 不要在这里做重采样，直接用 48kHz 采集（避免多次 resample 损失音质）
- 如果系统只能提供 44.1kHz，用高质量重采样（如 speex/rubato）上采样到 48kHz

---

### 2.2 预处理流水线（Project Mozart 核心）

**做什么**：把原始音频清洗成后端可用的标准流。后端不处理任何预处理，**前端必须做全**。

按顺序执行以下阶段（每帧 20ms 流过）：

| 阶段 | 功能 | 输入 | 输出 | 关键要求 |
|------|------|------|------|----------|
| **① 重采样/归一化** | 统一为 48kHz / mono / float32 / [-1,1] | 任意格式 | 48kHz mono f32 | 因果重采样，不可用整段 FFT |
| **② 瞬态抑制** | 检测键盘敲击、风扇点击，衰减突发噪声 | 48kHz f32 | 48kHz f32 | 只依赖过去+当前样本 |
| **③ AEC（回声消除）** | 消除扬声器回放到麦克风的回声 | mic + 远端参考 | 残差帧 | 远端参考来自系统回放环回 |
| **④ 去噪** | 主要噪声抑制 | 残差帧 | 增强帧 | **双档可选**：RNNoise（速度）/ DeepFilterNet（质量） |
| **⑤ 轻量去混响** | 减少房间混响 | 增强帧 | 干声帧 | 因果版，不倒序处理 |
| **⑥ AGC + 削波** | 自动增益到 -23 LUFS，硬限幅防削波 | 干声帧 | 增益规整帧 | 目标电平一致，不爆音 |
| **⑦ VAD + 打标** | 检测语音/静音，分段编号 | 增益帧 | + 元数据 | 每帧输出 vad_flag、segment_id、energy_db |

**设计文档参考**：预处理详细设计见 `docs/preprocessing-design.md`（由 Mozart 团队维护）。

**关键约束**：
- **全部因果**：每个模块只依赖过去和当前 20ms 帧，不允许看未来
- **端到端延迟 ≤ 25ms**：预处理 5ms + 网络 5ms + 后端推理 5ms + 播放缓冲 10ms
- **单帧处理预算 < 5ms**：否则会累积延迟

---

### 2.3 降采样到 16kHz（契约标准）

**做什么**：预处理输出契约流需要 **16kHz / mono / float32**，RVC 后端的特征提取直接消费 16kHz。

**技术细节**：
- 在 48kHz 做完所有预处理后，最后一步降采样到 16kHz
- 降采样算法：因果的，如 libsamplerate 的 SRC_SINC_FASTEST 或 speex resampler
- 每帧输出：16kHz × 20ms = **320 个 float32 样本**
- 范围：[-1.0, 1.0]，超出部分先在前端截断

**为什么不是 48kHz 直接发？**
- RVC 的特征提取器（HuBERT）只认 16kHz
- 16kHz 传输带宽更低（1300 bytes/帧 vs 3860 bytes/帧）
- 设计文档明确标准契约为 16kHz

---

### 2.4 契约封装 + UDP 发送（网络层）

**做什么**：把 16kHz 预处理帧 + 元数据打包成 UDP 定长包，发送到后端 `IP:18000`。

**包格式**（C 结构，小端序）：

```c
struct ContractAudioPacket {
    uint32_t magic;        // 0x4D5A5254 = 'MZRT'
    uint64_t pts_ns;       // 纳秒时间戳（uint64_t）
    uint32_t frame_idx;    // 单调递增帧序号（从 0 开始）
    uint8_t  vad_flag;     // 1 = 语音，0 = 静音
    uint8_t  energy_db;    // 能量分贝，映射到 0~255
    uint8_t  conf;         // 去噪置信度，0~255
    uint8_t  segment_id;   // 语音段编号，0 = 静音
    float    samples[320]; // 16kHz / 20ms / mono / float32
};
// 总大小：20 + 320×4 = 1300 bytes
```

**发送要求**：

| 项 | 要求 | 说明 |
|----|------|------|
| 目标地址 | `后端IP:18000` | UDP，无连接 |
| 发送间隔 | **严格 20ms ± 2ms** | 不能快也不能慢 |
| 帧序号 | 单调递增，不可跳号 | 后端用 frame_idx 检测丢包 |
| 时间戳 | 纳秒级，单调递增 | 用于音视频同步 |
| VAD 标记 | 静音帧必须填 0 | 后端会跳过推理，节省 GPU |
| 首次发送 | 后端收到第一帧后记录源地址 | 之后自动回发变声音频到该地址 |

**energy_db 编码公式**：

```c
uint8_t encode_energy_db(float raw_db) {
    // raw_db 典型范围：-96 dB (静音) ~ 0 dB (满幅)
    float clamped = fmaxf(-96.0f, fminf(0.0f, raw_db));
    return (uint8_t)(clamped + 96.0f);  // 映射到 0~255
}
```

---

### 2.5 接收变声音频 + 播放（播放器）

**做什么**：从后端 `18000` 端口接收变声后的契约帧，解析出 48kHz 音频，送入声卡播放。

**接收包格式**：与发送包结构完全相同，但 Payload 更大

```c
struct ContractAudioPacket {
    // Header: 20 bytes（同上）
    float    samples[960];  // 48kHz / 20ms / mono / float32
};
// 总大小：20 + 960×4 = 3860 bytes
```

**播放流程**：

1. 从 UDP socket 接收回包（后端会自动回发到前端源地址）
2. 校验魔数 `0x4D5A5254`
3. 按 `frame_idx` 排序（如果乱序）
4. 提取 960 个 float32 样本
5. 重采样到本地声卡采样率（通常是 48kHz，直接播放）
6. 送入声卡缓冲区播放

**播放缓冲（Jitter Buffer）**：
- 建议维护 **2~3 帧缓冲**（40~60ms）
- 后端偶尔推理慢一帧时，播放器不卡顿
- 用 `frame_idx` 检测缺失帧，插静音或重复前一帧

---

## 3. 前端需要维护的状态

| 状态 | 类型 | 说明 |
|------|------|------|
| `frame_idx` | `uint32_t` | 发送帧序号，每帧 +1，溢出后回绕（后端按差值处理） |
| `segment_id` | `uint8_t` | 当前语音段编号，VAD 从 0→1 时 +1，静音时保持 |
| `pts_ns` | `uint64_t` | 纳秒级时间戳，可基于 `steady_clock` 或音频设备时钟 |
| `jitter_buffer` | `deque<Packet>` | 接收端排序缓冲，2~3 帧深度 |
| `backend_addr` | `sockaddr_in` | 后端 UDP 地址，初始化时配置 |

---

## 4. 前端不需要做什么

以下工作**全部由后端负责**，前端不要重复：

| ❌ 不要做 | 原因 |
|----------|------|
| 任何变声/音色转换 | 这是后端 RVC 的工作 |
| 再次去噪/降噪 | 前端预处理已经做完，后端不再处理 |
| 回声消除的后续处理 | AEC 在前端完成 |
| 重采样到 48kHz 给后端 | 后端输出已经是 48kHz，前端只负责播放 |
| 模型加载/切换 | 通过 HTTP API `18080` 调用后端 |

## 5. 与后端的 HTTP 控制交互（端口 18080）

前端除了 UDP 音频流，还需要通过 HTTP 与后端通信：

### 5.1 后端健康检查（启动时）

```bash
GET http://后端IP:18080/health

响应：{"status":"ok"}
```

前端在启动时轮询，确认后端在线后才开始发送音频。

### 5.2 获取后端状态（可选）

```bash
GET http://后端IP:18080/status

响应：{
  "mode": "mock",
  "latency_stats_ms": {"count":1234, "avg_ms":2.5, "max_ms":8.1},
  "contract_config": {
    "input_sample_rate_hz": 16000,
    "output_sample_rate_hz": 48000,
    "frame_duration_ms": 20
  }
}
```

用于前端确认后端配置与自身一致（采样率、帧长等）。

### 5.3 切换 RVC 模型（用户选择新音色时）

```bash
POST http://后端IP:18080/models/speaker_b/activate

响应：{"status":"activated", "model_id":"speaker_b"}
```

前端在用户界面提供模型选择，调用此接口切换。无需中断音频流。

### 5.4 列出可用模型（UI 下拉菜单）

```bash
GET http://后端IP:18080/models

响应：{"models": [{"id":"speaker_a", "exists":true, "loaded":false, "current":false}]}
```

## 6. 对接流程（按顺序）

```
Step 1: 确认后端在线
  └─► 前端发送 GET /health 到 18080，确认 {"status":"ok"}

Step 2: 检查配置一致性
  └─► 前端发送 GET /status，确认：
      - input_sample_rate_hz = 16000
      - frame_duration_ms = 20
      - output_sample_rate_hz = 48000

Step 3: 启动音频采集
  └─► 打开麦克风，48kHz / mono / float32，20ms 回调

Step 4: 启动预处理流水线
  └─► 按 §2.2 顺序处理每帧，最后降采样到 16kHz

Step 5: 启动 UDP 发送线程
  └─► 每 20ms 打包一帧，发送到 后端IP:18000
      帧序号从 0 开始单调递增

Step 6: 启动 UDP 接收线程
  └─► 从同一 socket 接收后端回包，解析 48kHz 音频

Step 7: 启动播放线程
  └─► 维护 2~3 帧 jitter buffer，排序后送入声卡

Step 8: 用户切换模型时
  └─► 调用 POST /models/{id}/activate，无需停止音频流
```

## 7. 常见坑与解决方案

| 问题 | 原因 | 解决 |
|------|------|------|
| 后端不回发音频 | 首次发送后后端才记录源地址 | 检查第一帧是否送达，魔数是否正确 |
| 播放有爆音/卡顿 | 帧间隔不严格 20ms | 用高精度定时器（如 `std::chrono::steady_clock` + 忙等补偿） |
| 变声音调不对 | 采样率不匹配 | 确认发送是 16kHz，后端输出是 48kHz，播放也是 48kHz |
| 延迟越来越大 | 前端发送快于后端处理 | 严格 20ms 间隔，后端推理慢时前端会自然丢包（frame_idx 检测） |
| 静音时 GPU 占用还高 | VAD 标记没正确填 0 | 静音帧 vad_flag 必须 = 0，后端才会 bypass |
| 输出包 3860 bytes 超过 MTU | 广域网部署 | 当前设计适合局域网/Jetson 本地；广域网需要拆帧或改用 TCP |

## 9. 检查清单（前端自测）

- [ ] 能独立运行预处理，输出 16kHz / float32 / 无爆音
- [ ] 静音帧 VAD 标记为 0，语音帧为 1
- [ ] 能发送 1300 bytes UDP 包到后端，魔数正确
- [ ] 帧序号严格递增，间隔 20ms ± 2ms
- [ ] 能接收后端回包（3860 bytes），解析出 960 个 float32
- [ ] 播放输出无卡顿、无延迟累积
- [ ] 能调用 `/health` 确认后端在线
- [ ] 能调用 `/models` 获取模型列表
- [ ] 切换模型时音频流不中断
- [ ] 长时间运行（>10分钟）内存和延迟稳定
