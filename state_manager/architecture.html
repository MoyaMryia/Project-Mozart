在同进程运行（如物理设备端到端处理）时，预处理和推理后端通过预分配的SPSC（单写单读）无锁环形队列实现极速传输：

  • 无锁更新：内部使用 C++11 std::atomic 读写索引，仅通过原子操作增减位置，不需要传统 mutex 锁保护，完全杜绝由于线程竞争锁而引起的音频爆音 (XRun)。
  • 内存对齐：传输的 C-ABI 结构体强制 #pragma pack(push, 1) 对齐。
    • raw_ring 运载 mozart_raw_frame_t (48k/20ms/3856B)
    • input_ring 运载 mozart_input_frame_t (16k/20ms/1296B)
    • output_ring 运载 mozart_output_frame_t (48k/20ms/3856B)

在分布式网络部署时，输入采集端与推理服务器之间通过 UDP 契约协议 (MZRT) 发送网络包：

  • 端口绑定：固定在 UDP 端口 18000 通信，定长格式以避免粘包。
  • 网络包大小设计：
    • 输入包 (16kHz)：20字节头部 + 320点浮点PCM = 1300字节 (安全适配网络 MTU 1500，防 IP 分片丢失)。
    • 输出包 (48kHz)：20字节头部 + 960点浮点PCM = 3860字节。
  • 安全校验：开头包含 4 字节魔数 0x4D5A5254 (即 ASCII 'MZRT')，后面跟随 16 字节元数据和浮点 PCM 音频数据。
// 网络封包示例 (IO/src/udp_stream.cpp) void pack_output_packet(const mozart_output_frame_t& in, OutputPacket& out) { write_u32_le(out.data(), MOZART_PACKET_MAGIC); write_u64_le(out.data() + 4, in.meta.pts_ns); write_u32_le(out.data() + 12, in.meta.frame_idx); out[16] = in.meta.vad_flag; out[17] = in.meta.energy_db; std::memcpy(out.data() + 20, in.pcm, 960 * sizeof(float)); }
控制与编排原理 (Control Plane)

状态管理器如何进行编排？

状态机遵循控制面与数据面解耦的设计。state_manager 本身不触碰音频数据指针，不执行核心的运算。它通过调用三大层级的**生命周期门面 (Lifecycle Facades)** 间接指挥全链路：

用户切换模式 (POST /api/mode/switch) │ ▼ [state_manager 锁判定与排队] ├─ 1. 模型大类切换 (Model Facade) │ └── pipeline->switch_model("alice") (载入/置换 CUDA 权重) │ ├─ 2. I/O 链路启停 (IO Facade) │ ├── mozart_io_create_pipewire_stream(Capture/Playback) │ └── mozart_io_open_stream() (分配无锁环，独占/释放物理设备) │ └─ 3. 后台线程控制 (Worker Facade) └── AudioWorker::start() / stop() (拉起/终止环形队列的消费)

为什么采用这种编排设计？

  • 极高稳定性：如果推理后端因并发任务阻塞或超时，状态机仍能独立接管 I/O 并下发静音填充，避免声卡死锁爆音。
  • 模型与驱动解耦：以后新增如 Seed-VC 等零样本变声模型时，只需更新 Model 与 Worker 面板，I/O 层的物理设备与网络驱动完全不需要改动。
